# Requirements Document

## Introduction

The Dummy Payment feature adds a simulated payment workflow to the Bestay booking application. It allows guests to "pay" for their bookings using mock payment methods (bank transfer, e-wallet, credit card) without integrating a real payment gateway. The feature introduces a Payment entity linked to each Booking, tracks payment lifecycle (pending, paid, failed, expired, refunded), and ties booking confirmation to successful payment. Admins can view and optionally verify payments. This feature is intended for academic and demonstration purposes and MUST NOT be used to process real financial transactions.

## Glossary

- **Payment_System**: The subsystem responsible for creating, processing, and tracking dummy payments linked to bookings.
- **Payment**: A record representing a single payment attempt for a Booking. Contains amount, method, status, reference code, timestamps, and optional proof data.
- **Payment_Method**: The dummy method selected by the Guest. Allowed values are `bank_transfer`, `e_wallet`, and `credit_card`.
- **Payment_Status**: The lifecycle state of a Payment. Allowed values are `pending`, `paid`, `failed`, `expired`, and `refunded`.
- **Payment_Reference**: A unique human-readable code (e.g., `PAY-YYYYMMDD-XXXXXX`) generated by the Payment_System to identify a Payment.
- **Payment_Expiry_Window**: A fixed duration of 60 minutes after Payment creation during which the Guest must complete the dummy payment.
- **Booking**: An existing entity representing a room reservation by a Guest. Has `status` values `pending`, `confirmed`, `cancelled`, `completed`.
- **Guest**: An authenticated User who owns a Booking and initiates Payment.
- **Admin**: An authenticated User with the admin role who can view and manage all Payments.
- **Booking_Service**: The existing `App\Services\BookingService` responsible for booking lifecycle operations.
- **Payment_Service**: A new service class responsible for Payment lifecycle operations.
- **Notification_Service**: The existing `App\Services\NotificationService` responsible for creating in-app notifications.
- **Simulated_Outcome**: The result of a dummy payment, determined by the Guest's explicit selection (`success` or `fail`) on the payment confirmation form.

## Requirements

### Requirement 1: Create Payment On Booking

**User Story:** As a Guest, I want a Payment record to be created automatically when I make a Booking, so that I know how much to pay and can complete the payment before the Booking is confirmed.

#### Acceptance Criteria

1. WHEN a Guest's Booking is successfully persisted during Booking creation, THE Payment_System SHALL create exactly one Payment record associated with that Booking within the same database transaction.
2. WHEN a Payment is created, THE Payment_System SHALL set the Payment `amount` equal to the associated Booking `total_price` with exact decimal equality and no rounding or currency conversion.
3. WHEN a Payment is created, THE Payment_System SHALL set the Payment `status` to `pending`.
4. WHEN a Payment is created, THE Payment_System SHALL generate a Payment_Reference that is unique across all existing Payment records and store it on the Payment record.
5. WHEN a Payment is created, THE Payment_System SHALL set the Payment `expires_at` to exactly 60 minutes (the Payment_Expiry_Window) after the Payment `created_at` timestamp.
6. WHEN a Booking is created, THE Booking_Service SHALL set the initial Booking `status` to `pending` instead of `confirmed`.
7. IF Payment creation fails for any reason during Booking creation, THEN THE Payment_System SHALL roll back the enclosing database transaction such that neither the Booking record nor any partial Payment record is persisted.
8. IF Payment creation fails during Booking creation, THEN THE Payment_System SHALL return an error response to the Guest indicating that the Booking could not be completed and no charge is owed.

### Requirement 2: Select Dummy Payment Method

**User Story:** As a Guest, I want to choose from multiple dummy payment methods, so that the simulated flow resembles a real checkout experience.

#### Acceptance Criteria

1. WHEN a Guest opens a Payment with `status` equal to `pending`, THE Payment_System SHALL display exactly three selectable Payment_Method options: `bank_transfer`, `e_wallet`, and `credit_card`.
2. WHEN a Guest submits a Payment_Method selection for a Payment with `status` equal to `pending` and the submitted value is exactly one of `bank_transfer`, `e_wallet`, or `credit_card`, THE Payment_System SHALL persist the selected Payment_Method on the Payment record and return a confirmation response identifying the persisted Payment_Method value.
3. IF a Guest submits a Payment_Method selection where the submitted value is missing, empty, null, or not exactly one of `bank_transfer`, `e_wallet`, or `credit_card`, THEN THE Payment_System SHALL reject the submission, leave the previously stored Payment_Method value unchanged, and return a validation error identifying the invalid field and listing the allowed values (`bank_transfer`, `e_wallet`, `credit_card`).
4. WHILE a Payment `status` is one of `paid`, `failed`, `expired`, or `refunded`, IF a Guest submits a Payment_Method change for that Payment, THEN THE Payment_System SHALL reject the submission, leave the stored Payment_Method unchanged, and return an error response indicating that the Payment_Method cannot be changed given the current Payment `status`.

### Requirement 3: Process Dummy Payment

**User Story:** As a Guest, I want to simulate completing the payment and choose whether it succeeds or fails, so that I can demonstrate both success and failure flows in the application.

#### Acceptance Criteria

1. WHEN a Guest submits a payment confirmation with Simulated_Outcome `success` for a `pending` Payment, THE Payment_System SHALL set the Payment `status` to `paid` and set `paid_at` to the current UTC timestamp with second-level precision.
2. WHEN a Payment transitions to `paid`, THE Booking_Service SHALL update the associated Booking `status` to `confirmed`.
3. WHEN a Guest submits a payment confirmation with Simulated_Outcome `fail` and a non-empty failure reason of at most 500 characters for a `pending` Payment, THE Payment_System SHALL set the Payment `status` to `failed` and store the Guest-provided failure reason on the Payment record.
4. WHILE a Payment `status` is `failed`, IF the associated Booking `status` is `pending` and the Booking has fewer than 5 total Payment attempts, THEN THE Payment_System SHALL allow the Guest to create a new Payment attempt for the same Booking.
5. IF a Guest submits a payment confirmation for a Payment whose `status` is not `pending`, THEN THE Payment_System SHALL reject the submission without modifying the Payment or the associated Booking, and return an error message identifying the current Payment `status`.
6. WHEN a Payment transitions to `paid`, THE Payment_System SHALL apply the Payment `status` change and the associated Booking `status` change atomically within a database transaction, retrying the transaction up to 2 times on transient failure before returning an error, such that if the final attempt fails, neither change is persisted.
7. IF a Guest submits a payment confirmation with a Simulated_Outcome value other than `success` or `fail`, THEN THE Payment_System SHALL reject the submission without modifying the Payment, and return an error message indicating the Simulated_Outcome is invalid.
8. IF a Guest submits a payment confirmation with Simulated_Outcome `fail` and a failure reason that is empty or exceeds 500 characters, THEN THE Payment_System SHALL reject the submission without modifying the Payment, and return an error message indicating the failure reason is invalid.
9. THE Payment_System SHALL NOT allow a Payment to transition backwards from `paid` or `failed` status to `pending` status, and SHALL reject any such request with an error indicating that backward transitions are not permitted.

### Requirement 4: Payment Expiry

**User Story:** As an Admin, I want unpaid Payments to expire automatically after the Payment_Expiry_Window, so that rooms are not held indefinitely by Guests who abandon checkout.

#### Acceptance Criteria

1. WHEN a Guest views a `pending` Payment whose `expires_at` timestamp is earlier than the current system time, THE Payment_System SHALL transition the Payment `status` from `pending` to `expired` and persist the change before rendering the view.
2. WHEN a Payment transitions to `expired`, THE Booking_Service SHALL update the associated Booking `status` to `cancelled` only if the Booking `status` is currently `pending`, and SHALL leave the Booking `status` unchanged for any other Booking state.
3. IF a Guest submits a payment confirmation for a Payment whose `expires_at` timestamp is earlier than the current system time, THEN THE Payment_System SHALL reject the submission without recording the confirmation, transition the Payment `status` from `pending` to `expired`, and return an error response indicating that the Payment has expired.
4. WHERE a scheduled task is configured, THE Payment_System SHALL execute the expiry job at an interval not exceeding 5 minutes and transition every `pending` Payment whose `expires_at` is earlier than the current system time to `expired` without requiring Guest interaction.
5. WHEN a Payment is created, THE Payment_System SHALL set the Payment `expires_at` to the Payment `created_at` plus the Payment_Expiry_Window of 60 minutes. IF the system detects that the computed `expires_at` would be earlier than `created_at` (e.g., due to clock adjustment), THE Payment_System SHALL adjust `expires_at` to be `created_at` plus the Payment_Expiry_Window.
6. WHEN a Payment transitions to `expired`, THE Notification_Service SHALL send the associated Guest an in-app notification indicating that the Payment has expired and the Booking has been cancelled.
7. WHEN a Booking is cancelled by any process while its associated Payment `status` is `pending`, THE Payment_System SHALL transition that Payment `status` to `expired` within 5 seconds of the cancellation being committed.

### Requirement 5: View Payment Details And History

**User Story:** As a Guest, I want to view the Payment details and history for my Bookings, so that I can track which Bookings are paid and retrieve Payment_Reference codes.

#### Acceptance Criteria

1. WHEN an authenticated Guest requests the Payment list for their own Bookings, THE Payment_System SHALL return every Payment whose associated Booking `user_id` equals the authenticated Guest `id`, ordered by `created_at` descending, and SHALL paginate the result set at a maximum of 20 Payments per page.
2. WHEN an authenticated Guest requests Payment details for a specific Payment owned by one of the Guest's own Bookings, THE Payment_System SHALL return the Payment `amount`, `status`, `method`, `reference`, `created_at`, `paid_at`, and `expires_at` fields.
3. IF a Guest requests a Payment that does not belong to any of the Guest's own Bookings, THEN THE Payment_System SHALL return a 403 Forbidden response and SHALL NOT include any Payment data in the response body.
4. IF a Guest requests Payment details using a Payment identifier that does not exist, THEN THE Payment_System SHALL return a 404 Not Found response indicating the Payment was not found and SHALL NOT return any Payment data.
5. IF a request is made to any Payment list or Payment detail endpoint by an unauthenticated requester, THEN THE Payment_System SHALL reject the request with a 401 Unauthorized response and SHALL NOT return any Payment data.
6. IF a request is made for the Admin Payment list by a requester that does not have the Admin role, THEN THE Payment_System SHALL return a 403 Forbidden response and SHALL NOT return any Payment records.
7. WHEN an Admin requests the Payment list, THE Payment_System SHALL return all Payments across all Bookings ordered by `created_at` descending, paginated at a maximum of 20 Payments per page, and SHALL restrict the result set to Payments that match every supplied filter value when any combination of the optional `status`, `method`, and `booking_id` parameters is provided.
8. IF an Admin supplies a `status` or `method` filter value that is not one of the system's defined enumerated values for that field, THEN THE Payment_System SHALL return a validation error response indicating the invalid filter and SHALL NOT return any Payment records.

### Requirement 6: Refund On Booking Cancellation

**User Story:** As a Guest, I want my Payment to be marked as refunded when a paid Booking is cancelled, so that the Payment history accurately reflects the Booking lifecycle.

#### Acceptance Criteria

1. WHEN a Booking with an associated Payment whose `status` is `paid` is cancelled, THE Payment_System SHALL, within 5 seconds of the cancellation event being committed, update the Payment `status` to `refunded`, set `refunded_at` to the server UTC timestamp of the cancellation event, leave all other Payment fields unchanged, and return a success response confirming the new `status` and `refunded_at` values.
2. WHEN a Booking with an associated Payment whose `status` is `pending` is cancelled, THE Payment_System SHALL, within 5 seconds of the cancellation event being committed, update the Payment `status` to `expired`, leave `refunded_at` unset, leave all other Payment fields unchanged, and return a success response confirming the new `status`.
3. IF a cancellation is requested for a Booking whose associated Payment `status` is already a terminal status (`refunded`, `expired`, or `failed`), THEN THE Payment_System SHALL reject the status transition, leave the Payment record unchanged, and return an error response indicating that the Payment is in a terminal status and naming the current terminal `status` value.
4. IF a Booking cancellation is processed and the Booking has no associated Payment record, THEN THE Payment_System SHALL perform no Payment status change, leave all Payment records unchanged, and return a response indicating that no Payment was linked to the Booking.
5. IF the same Booking cancellation event is received more than once for a Booking whose Payment has already been transitioned by any prior cancellation attempt for that Booking, THEN THE Payment_System SHALL leave the Payment `status` and `refunded_at` unchanged and return a success response reflecting the existing Payment state.

### Requirement 7: Admin Payment Verification

**User Story:** As an Admin, I want to manually mark a Payment as paid or failed, so that I can resolve edge cases where a Guest reports a problem with the dummy flow.

#### Acceptance Criteria

1. WHEN an authenticated Admin submits a manual Payment_Status update for a `pending` Payment with a target status of `paid` or `failed`, THE Payment_System SHALL apply the state transition logic defined in Requirement 3 and persist the updated Payment_Status within 2 seconds of receiving the request.
2. IF a non-Admin User submits a manual Payment_Status update, THEN THE Payment_System SHALL reject the request with a 403 Forbidden response and SHALL NOT modify the Payment_Status.
3. IF an Admin submits a manual Payment_Status update with a target status that is not one of `paid`, `failed`, or `refunded`, THEN THE Payment_System SHALL reject the submission, return a validation error indicating the allowed target values (`paid`, `failed`, `refunded`), and SHALL NOT modify the Payment_Status.
4. WHEN an Admin submits a manual Payment_Status update with target status `refunded`, THE Payment_System SHALL require the Payment's current status to be `paid` before persisting the transition. Refunds SHALL only be allowed for Payments with `paid` status.
5. IF an Admin submits a manual Payment_Status update with target status `refunded` while the Payment's current status is not `paid` or the associated Booking `status` is not `cancelled`, THEN THE Payment_System SHALL reject the request, return an error indicating the unmet precondition, and retain the existing Payment_Status.
6. WHEN an Admin manual Payment_Status update is successfully persisted, THE Payment_System SHALL record an audit entry capturing the Admin identifier, the previous status, the new status, and the update timestamp before returning the success response.

### Requirement 8: Payment Status Transition Rules

**User Story:** As a developer, I want Payment status transitions to follow explicit rules, so that the Payment lifecycle is predictable and auditable.

#### Acceptance Criteria

1. THE Payment_System SHALL allow a Payment_Status transition only if the (source, target) pair is one of the following: (`pending`, `paid`), (`pending`, `failed`), (`pending`, `expired`), (`paid`, `refunded`).
2. IF a requested Payment_Status transition has a (source, target) pair not listed in Acceptance Criterion 1, THEN THE Payment_System SHALL reject the transition, leave the Payment_Status unchanged, and return an error response indicating an invalid transition and identifying both the current source status and the requested target status.
3. IF a Payment_Status transition is requested on a Payment whose current `status` is `failed`, `expired`, or `refunded`, THEN THE Payment_System SHALL reject the requested transition, leave the Payment_Status unchanged, and return an error response indicating that the Payment is in a terminal status and identifying that terminal status.
4. WHEN a valid Payment_Status transition is applied, THE Payment_System SHALL persist the Payment_Status update together with any Booking `status` update required by Requirement 3 or Requirement 6 as a single atomic unit, such that either both updates are committed or neither is committed. The system SHALL only reject transitions when atomic persistence fails as described in Acceptance Criterion 5.
5. IF the atomic persistence described in Acceptance Criterion 4 fails for any reason, THEN THE Payment_System SHALL roll back both the Payment_Status update and the associated Booking `status` update, leave both records in their pre-transition values, and return an error response indicating that the transition was not applied.
6. WHEN a Payment_Status transition is committed, THE Payment_System SHALL record an audit entry containing the source status, the target status, the identifier of the actor that requested the transition, and a transition timestamp with second-level precision.

### Requirement 9: Payment Data Integrity

**User Story:** As a developer, I want Payment data to be structurally consistent, so that reports and the UI can rely on invariants.

#### Acceptance Criteria

1. THE Payment_System SHALL enforce a one-to-many relationship between a Booking and its Payments, such that for each Booking, the count of Payments with `status` equal to `pending` or `paid` is at most 1 at any single point in time.
2. THE Payment_System SHALL store the Payment `amount` as a decimal with precision 10 and scale 2, accepting values from 0.01 to 99999999.99 inclusive, rounded to 2 decimal places using half-up rounding.
3. THE Payment_System SHALL ensure every `Payment_Reference` value is unique across the Payment table, with a length between 1 and 64 characters inclusive.
4. IF an attempt is made to create a second concurrent Payment with `status` equal to `pending` for a Booking that already has a Payment with `status` equal to `pending` or `paid`, THEN THE Payment_System SHALL reject the new Payment based on the final committed state of the existing Payment record, return a conflict error indicating the booking already has an active payment, and preserve the existing Payment record unchanged. Concurrent attempts during the processing window before the first payment is committed SHALL be allowed and resolved at commit time.
5. IF an attempt is made to create or update a Payment with an `amount` that is null, less than 0.01, greater than 99999999.99, or non-numeric, THEN THE Payment_System SHALL reject the operation, return a validation error indicating the amount is invalid, and persist no changes to the Payment table.
6. IF an attempt is made to create a Payment with a `Payment_Reference` value that already exists in the Payment table, THEN THE Payment_System SHALL reject the operation, return a uniqueness violation error indicating the reference is duplicated, and persist no changes to the Payment table.

### Requirement 10: Notifications

**User Story:** As a Guest, I want to receive in-app notifications when my Payment status changes, so that I am informed about the outcome without refreshing the page manually.

#### Acceptance Criteria

1. WHEN a Payment transitions to `paid`, THE Notification_Service SHALL create an in-app Notification for the associated Guest within 5 seconds of the transition being committed, with initial read status set to unread, and containing the Booking identifier and Payment_Reference.
2. WHEN a Payment transitions to `failed` or `expired`, THE Notification_Service SHALL create an in-app Notification for the associated Guest within 5 seconds of the transition being committed, with initial read status set to unread, and containing the Booking identifier and the new Payment_Status.
3. WHEN a Payment transitions to `refunded`, THE Notification_Service SHALL create an in-app Notification for the associated Guest within 5 seconds of the transition being committed, with initial read status set to unread, and containing the Booking identifier and the refund timestamp.
4. IF Notification creation fails on the first attempt, THEN THE Notification_Service SHALL retry creation up to 3 times with a minimum interval of 2 seconds between attempts, SHALL NOT block or reverse the Payment_Status transition regardless of whether notification creation ultimately succeeds or fails, and SHALL log the failure for administrator visibility if all retries are exhausted. The Payment_Status transition SHALL complete even if all notification retries are exhausted.
5. WHEN a Guest requests their Notification list, THE Notification_Service SHALL return Notifications ordered by `created_at` descending, with each entry indicating its read/unread status.
